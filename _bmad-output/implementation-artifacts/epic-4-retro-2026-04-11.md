## ⚠️ What Didn't Go Well（需要改进的方面）

### 1. 技术选型缺乏深度调研

**问题描述：**

在 Story 4-1 中，我们选择了 **Tiptap** 作为 Markdown 编辑器内核，但这个选择存在以下问题：

**Charlie (Senior Dev):** "Tiptap 是一个强大的富文本编辑器框架，但我们只需要一个简单的 Markdown 编辑器。这就像用大炮打蚊子——过度工程化了。"

**具体问题：**
1. **学习曲线陡峭：** Elena 花了大量时间学习 Tiptap 的概念（Schema、Commands、Extensions）
2. **非原生 Markdown：** Tiptap 内部使用 HTML，需要额外的转换层（turndown.js）才能输出 Markdown
3. **集成复杂度高：** 需要配置多个扩展包（StarterKit、Image、CodeBlock 等）
4. **开发时间翻倍：** 预计 6 小时，实际 12+ 小时

**Elena (Junior Dev):** "作为初级开发，我在理解 Tiptap 的 API 时遇到了很大困难。如果选择一个更简单的方案（如 ByteMD 或 Vditor），我可以更快交付价值。"

**根因分析：**
- ❌ 没有对比其他方案（ByteMD、Vditor、simplemde）
- ❌ 没有评估团队技能匹配度（团队对 Tiptap 不熟悉）
- ❌ 没有考虑 MVP 阶段的实际需求（只需要基础编辑功能）
- ❌ 缺少 PoC（概念验证）环节

**影响：**
- 开发时间增加 100%（6h → 12h+）
- 团队学习成本高，影响后续 Story 的开发速度
- 引入了不必要的依赖（~100KB 包体积）

---

### 2. 缺少 PoC 验证环节

**问题描述：**

在引入 Tiptap 之前，我们没有做任何概念验证（PoC），直接在生产环境中集成，导致发现问题后难以回退。

**Charlie (Senior Dev):** "如果我们提前做一个 2 小时的 PoC，就能发现 Tiptap 不是原生的 Markdown 编辑器，需要额外的 HTML ↔ Markdown 转换层。这个发现可以让我们重新评估技术选型。"

**具体问题：**
1. **没有验证核心技术可行性：** Tiptap 能否满足我们的 Markdown 编辑需求？
2. **没有评估集成难度：** 与 Vue 3 + TypeScript 的兼容性如何？
3. **没有性能测试：** 编辑器初始化速度、输入延迟是否符合预期？
4. **没有团队试用：** 其他成员是否容易上手？

**根因分析：**
- ❌ 急于开始编码，跳过了验证环节
- ❌ 过于自信，认为 Tiptap 是成熟方案，不需要验证
- ❌ 没有意识到 PoC 的价值（2 小时 PoC 可以节省 6 小时返工）

**影响：**
- 发现 HTML ↔ Markdown 转换问题后，已经投入了大量时间
- 回退成本高，只能继续使用 Tiptap
- 技术债务积累（转换层的维护成本）

---

### 3. 基础设施决策滞后

**问题描述：**

图片存储方案没有在架构设计阶段确定，而是等到实现时才决定使用本地文件系统。

**Dana (QA Engineer):** "从测试角度，我发现当前的图片存储方案存在隐患：随着内容增多，服务器磁盘空间会快速增长，而且没有 CDN 加速，图片加载速度慢。"

**具体问题：**
1. **本地存储的局限性：**
   - 服务器磁盘空间有限
   - 无法水平扩展（多服务器部署时图片不同步）
   - 没有备份和容灾机制

2. **缺少图片处理：**
   - 没有压缩（用户上传 5MB 图片，直接存储）
   - 没有格式转换（WebP 可以减小 30% 体积）
   - 没有生成缩略图（列表页加载大图浪费带宽）

3. **没有 CDN 加速：**
   - 图片加载速度慢，影响用户体验
   - 服务器带宽压力大

**根因分析：**
- ❌ 基础设施类决策（存储、CDN、缓存）未在 Architecture 文档中明确
- ❌ 只关注功能实现，忽略了可扩展性
- ❌ 认为"以后再说"，但后期变更成本高

**影响：**
- 后期可能需要迁移到云存储（阿里云 OSS、腾讯云 COS）
- 迁移成本高（数据迁移、代码重构、测试验证）
- 用户体验受损（图片加载慢）

### 4. 数据模型设计未考虑查询需求

**问题描述：**

在 Story 4-3（博客发布）中，标签使用 JSON 字段存储，导致后续难以搜索和统计。

**Alice (Product Owner):** "从产品角度，我担心当前的标签实现无法满足未来的需求。比如，我们想展示'热门标签'、'标签云'，或者让用户按标签筛选博客，JSON 字段很难高效支持这些查询。"

**具体问题：**
1. **无法高效搜索：** MySQL 对 JSON 字段的查询性能差
2. **无法统计：** 难以统计每个标签的使用次数
3. **数据一致性：** 容易出现标签拼写错误（"Java" vs "java" vs "JAVA"）
4. **无法关联：** 难以实现"相关文章推荐"（基于共同标签）

**根因分析：**
- ❌ 只考虑了写入便利性（JSON 字段简单）
- ❌ 没有预判未来的业务场景（标签搜索、统计、推荐）
- ❌ 没有咨询 DBA 或资深开发的意见

**影响：**
- 后期可能需要重构数据模型（JSON → 关系表）
- 迁移成本高（数据迁移、代码修改、测试验证）
- 功能受限（无法实现标签相关的高级功能）

## 💡 Key Insights（关键洞察）

### 1. MVP 阶段应该优先选择"开箱即用"的方案

**Charlie (Senior Dev):** "对于 MVP 阶段，我们应该优先选择**简单、开箱即用**的方案，而不是**高度可定制**的方案。等验证了需求后，再考虑重构为更复杂的方案。"

**原则：**
- **YAGNI（You Ain't Gonna Need It）：** 只实现当前需要的功能
- **KISS（Keep It Simple, Stupid）：** 保持简单
- **快速验证 > 技术完美：** MVP 的目标是验证业务假设，而非展示技术实力

**示例对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Tiptap | 高度可定制 | 学习曲线陡、非原生 Markdown | 需要复杂定制的富文本编辑器 |
| ByteMD | 开箱即用、轻量 | 定制性较弱 | MVP 阶段的 Markdown 编辑器 |
| Vditor | 功能丰富、中文友好 | 包体积较大 | 需要即时渲染模式的编辑器 |
| textarea + marked | 最简单 | 无实时编辑体验 | 极简场景 |

**结论：** 对于 Epic 4 的需求，ByteMD 是更好的选择。

---

### 2. 技术选型要考虑团队能力

**Elena (Junior Dev):** "技术选型不应该只看技术本身的优劣，还要考虑团队的技能水平和学习能力。如果一个技术方案很好，但团队需要花费大量时间学习，那它可能不是最佳选择。"

**评估维度：**
1. **团队熟悉度：** 团队成员是否有相关经验？
2. **学习成本：** 预计需要多少时间学习？
3. **文档质量：** 文档是否完善？是否有中文资源？
4. **社区活跃度：** 遇到问题时能否快速找到答案？
5. **调试难度：** 出现问题时是否容易定位和修复？

**改进建议：**
- 在 Sprint Planning 时，评估团队成员对新技术的熟悉程度
- 如果需要引入新技术，预留足够的学习时间（Spike）
- 准备备选方案（Plan B），以防主方案不可行

---

### 3. 基础设施决策要提前到架构设计阶段

**Dana (QA Engineer):** "基础设施类决策（存储、CDN、缓存、消息队列）应该在架构设计阶段就确定，因为这些决策的变更成本高，后期修改困难。"

**应该在 Architecture 文档中明确的内容：**
1. **文件存储方案：** 本地文件系统 / 云存储（OSS/COS）/ 分布式存储
2. **图片处理策略：** 压缩、格式转换、缩略图生成
3. **CDN 加速方案：** 是否使用 CDN？哪家服务商？
4. **缓存策略：** Redis / Memcached / 浏览器缓存
5. **消息队列：** 是否需要？RabbitMQ / Kafka / RocketMQ
6. **备份和恢复：** 数据备份频率、恢复策略

**原则：**
- **可扩展性优先：** 考虑未来 1-2 年的增长
- **运维成本可控：** 不要引入过于复杂的方案
- **抽象层设计：** 预留接口，便于后期替换（如 StorageService）

---

### 4. 数据模型设计要考虑查询需求

**Alice (Product Owner):** "设计数据模型时，不仅要考虑写入便利性，还要考虑未来的查询需求。如果一个字段将来需要搜索、过滤、统计，就应该使用独立表，而不是 JSON 字段。"

**决策框架：**

在设计数据模型时，问自己三个问题：
1. **这个字段将来需要搜索吗？**
2. **需要按这个字段过滤吗？**
3. **需要统计这个字段的分布吗？**

如果答案是 **Yes**，使用**独立表 + 关联表**。  
如果答案是 **No**，可以使用 **JSON 字段**。

**示例：**

```sql
-- ❌ 不好的设计：标签使用 JSON 字段
CREATE TABLE content (
    id BIGINT PRIMARY KEY,
    tags JSON  -- 难以搜索、统计
);

-- ✅ 好的设计：标签使用独立表
CREATE TABLE tags (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) UNIQUE NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE content_tags (
    content_id BIGINT,
    tag_id BIGINT,
    PRIMARY KEY (content_id, tag_id),
    FOREIGN KEY (content_id) REFERENCES content(id),
    FOREIGN KEY (tag_id) REFERENCES tags(id)
);

-- 可以轻松实现：
-- 1. 搜索包含某个标签的内容
SELECT c.* FROM content c
JOIN content_tags ct ON c.id = ct.content_id
JOIN tags t ON ct.tag_id = t.id
WHERE t.name = 'Java';

-- 2. 统计热门标签
SELECT t.name, COUNT(*) as count
FROM tags t
JOIN content_tags ct ON t.id = ct.tag_id
GROUP BY t.id
ORDER BY count DESC
LIMIT 10;
```

   - 服务商：阿里云 CDN
   - 缓存策略：图片缓存 30 天
   - 刷新策略：内容更新时主动刷新 CDN

4. **缓存策略：**
   - 热点数据：Redis（TTL 5 分钟）
   - 静态资源：浏览器缓存（Cache-Control）
   - 页面缓存：SSR + CDN

5. **备份和恢复：**
   - 数据库备份：每天凌晨 2 点全量备份
   - 文件备份：每周日全量备份
   - 恢复演练：每季度一次

## 🔮 Next Epic Preparation（下一个 Epic 的准备）

### Epic 5: 内容审核机制

基于 Epic 4 的经验教训，在开始 Epic 5 前，建议进行以下准备：

#### 1. 技术选型评估

**需要评估的技术方案：**
1. **审核队列的实时更新方案：**
   - WebSocket（实时推送）
   - SSE（Server-Sent Events）
   - 轮询（每 30 秒请求一次）

2. **投票机制的数据一致性保证：**
   - 数据库事务
   - 乐观锁
   - 分布式锁（Redis）

3. **审核记录的审计日志存储方案：**
   - 独立表（`audit_logs`）
   - Elasticsearch（便于搜索和分析）
   - 日志文件（简单但难以查询）

#### 2. PoC 验证

**需要进行 PoC 验证的内容：**
1. **WebSocket 与 Spring Boot 集成：**
   - 验证实时推送的可行性
   - 测试并发连接数
   - 评估服务器资源占用

2. **并发投票的场景测试：**
   - 模拟多个审核员同时投票
   - 验证数据一致性
   - 检测竞态条件

3. **审核流程的状态机设计：**
   - 定义状态流转规则（pending → approved/rejected）
   - 验证状态机的正确性
   - 测试异常场景（网络中断、超时）

#### 3. 基础设施准备

**需要提前确定的基础设施方案：**
1. **是否需要消息队列：**
   - 如果使用 WebSocket，可能需要 RabbitMQ/Kafka 处理审核通知
   - 评估消息队列的必要性和成本

2. **审核数据的备份策略：**
   - 审核记录的重要性高，需要定期备份
   - 考虑异地容灾

3. **权限控制的细粒度设计：**
   - 审核员只能审核分配给自己的内容
   - 管理员可以审核所有内容
   - 需要设计灵活的权限模型
