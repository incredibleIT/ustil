## 遇到的挑战

### 1. 并发竞态条件 - 登录失败计数
**问题：** 使用 `user.setLoginFailCount(user.getLoginFailCount() + 1)` 在并发场景下可能丢失更新

**影响：** 高并发时账号锁定机制可能失效

**解决方案：**
```java
// 使用 MyBatis-Plus 的 UpdateWrapper 实现原子更新
UpdateWrapper<User> updateWrapper = new UpdateWrapper<>();
updateWrapper.eq("id", user.getId())
    .setSql("login_fail_count = login_fail_count + 1");
userMapper.update(null, updateWrapper);
```

**教训：** 所有计数器更新都应使用数据库原子操作

### 2. NumberFormatException - JWT Principal 类型
**问题：** JwtAuthenticationFilter 中 `authentication.getPrincipal()` 返回 String，直接转 Long 导致异常

**根因分析：**
- JwtTokenProvider 生成 Token 时使用 `userId.toString()` (String)
- 过滤器解析后设置到 SecurityContext 的是 String
- SecurityUtils 中 `Long.parseLong(authentication.getName())` 可能失败

**解决方案：**
```java
// JwtAuthenticationFilter 中
String userIdStr = jwtTokenProvider.getUserIdFromToken(token);
// 直接使用 String，不转换为 Long
UsernamePasswordAuthenticationToken authToken = 
    new UsernamePasswordAuthenticationToken(userIdStr, null, authorities);
```

**教训：** JWT Payload 中的数据类型应保持一致，避免类型转换

### 3. 状态不一致 - Story 2-3 标记问题
**问题：** sprint-status.yaml 中 2-3 标记为 done，但故事文件状态为 ready-for-dev

**影响：** 冲刺状态跟踪不准确

**解决方案：** 
- 开发完成后立即同步更新故事文件状态
- 添加 completed 日期字段
- 定期校验 sprint-status.yaml 与故事文件一致性

**教训：** 建立开发完成后的状态更新检查清单

## 关键洞察

1. **安全性不能妥协** - 每个安全相关的细节都需要仔细审查
2. **并发问题容易被忽视** - 计数器更新、状态变更都要考虑并发场景
3. **状态同步很重要** - sprint-status.yaml、故事文件、实际代码状态必须一致
4. **代码审查价值巨大** - 发现并发竞态、类型转换、性能优化等多个问题
5. **前后端协同开发** - API 契约先行，减少联调时间
6. **自定义校验注解** - 提升代码可读性和可维护性，值得推广
