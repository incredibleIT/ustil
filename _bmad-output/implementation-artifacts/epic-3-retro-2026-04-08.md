## 遇到的挑战

### 1. 状态同步问题反复出现
**问题描述：** 
- Sprint-status.yaml 中的状态与实际 Story 文件状态不一致
- Story 3-2 标记为 review 但实际已完成
- Story 3-3 标记为 ready-for-dev 但实际已完成

**根本原因：**
- 开发完成后忘记同步更新 sprint-status.yaml
- 缺乏自动化的状态校验机制
- BMAD 工作流执行不够严格

**解决方案：**
- 建立开发完成后的状态更新检查清单
- 每次完成任务后立即更新 sprint-status.yaml
- 定期运行 `/bmad-sprint-status` 进行校验

**教训：** 状态同步是团队协作的基础，必须严格执行

### 2. MyBatis-Plus 自动填充配置缺失
**问题描述：**
- Story 3-5 中删除了 `notification.setCreatedAt(LocalDateTime.now())`
- 误以为 `@TableField(fill = FieldFill.INSERT)` 会自动填充
- 实际项目中没有配置 MetaObjectHandler

**影响：**
- 通知记录的 createdAt 字段为 null
- 接口测试时发现数据异常

**解决方案：**
- 恢复手动设置 createdAt 的代码
- 或者在后续 Epic 中添加 MyBatisPlusConfig 配置 MetaObjectHandler

**教训：** 不要假设框架行为，必须先验证配置是否存在

### 3. Flyway 迁移版本号冲突
**问题描述：**
- Story 3-5 创建了两个新的迁移文件（V6, V7）
- 但项目中已有 V6 和 V7 文件（来自之前的 Story）
- 导致 Flyway 启动失败：Found more than one migration with version 6

**影响：**
- 后端服务无法启动
- 需要重新命名迁移文件为 V8 和 V9

**解决方案：**
- 创建迁移文件前先检查现有版本号
- 使用命令：`ls backend/src/main/resources/db/migration/ | sort`
- 采用最大版本号 +1 的策略

**教训：** 数据库迁移版本号必须唯一且递增

### 4. Node 版本兼容性问题
**问题描述：**
- 前端使用 Vite 6.0.3，需要 Node >= 14.18
- 系统默认 Node 版本为 v10.24.1
- 启动时报错：SyntaxError: Unexpected token

**影响：**
- 前端服务无法启动
- 浪费了排查时间

**解决方案：**
- 使用 nvm 切换到 Node v20.20.2
- 在项目根目录添加 .nvmrc 文件指定 Node 版本

**教训：** 多版本管理工具（nvm）是前端开发的必备工具

### 5. SecurityUtils 调用方式混淆
**问题描述：**
- Story 3-5 中误用了静态调用：`SecurityUtils.getCurrentUserId()`
- 实际 SecurityUtils 需要依赖注入实例
- 编译报错：non-static method cannot be referenced from a static context

**影响：**
- 编译失败
- 需要修改为注入实例后调用

**解决方案：**
```java
// 错误用法
Long userId = SecurityUtils.getCurrentUserId();

// 正确用法
private final SecurityUtils securityUtils;
Long userId = securityUtils.getCurrentUserId();
```

**教训：** 工具类的调用方式需要先查看源码或文档确认

## 关键洞察

### 1. BMAD 工作流的价值与挑战
**洞察：** BMAD 提供了结构化的开发流程，但需要团队严格遵守才能发挥价值。

**证据：**
- ✅ 好处：Story 文档详细，Acceptance Criteria 清晰，减少了需求歧义
- ❌ 挑战：状态同步容易被忽视，导致 sprint-status.yaml 不准确
- 💡 建议：将状态更新作为 Definition of Done 的一部分

### 2. 代码审查应该更早介入
**洞察：** Story 3-5 在完成后才进行代码审查，发现问题需要返工。

**证据：**
- 发现了 7 个问题需要修复
- 如果边开发边审查，可以避免累积问题
- Review 阶段修复问题的成本比开发阶段高 3-5 倍

**建议：** 
- 采用增量式代码审查（每完成一个 Task 就审查）
- 使用 AI 辅助实时审查（如 Lingma 的即时反馈）

### 3. 技术债务需要主动管理
**洞察：** Epic-2 遗留的技术债务在 Epic-3 中仍未解决。

**证据：**
- Refresh Token 黑名单机制仍未实现
- 单元测试覆盖率不足
- 管理员操作日志未持久化

**建议：**
- 每个 Epic 预留 20% 时间用于偿还技术债务
- 建立技术债务看板，跟踪优先级

### 4. 前后端协同效率可以提升
**洞察：** 前后端联调时发现 API 契约理解不一致。

**证据：**
- 前端期望的响应格式与后端返回不完全匹配
- 部分字段命名不一致（camelCase vs snake_case）
- 错误码映射不完整

**建议：**
- 使用 OpenAPI/Swagger 定义 API 契约
- 前后端共同评审 API 设计后再开始开发
- 建立 Mock Server 支持并行开发

### 5. 通知系统的扩展性考虑不足
**洞察：** 当前通知系统只支持转正相关通知，未来可能需要扩展。

**证据：**
- NotificationType 常量类只有 2 种类型
- 通知内容硬编码在 Service 层
- 缺少通知模板机制

**建议：**
- 设计通用的通知模板引擎
- 支持多种通知渠道（站内信、邮件、短信）
- 添加通知偏好设置（用户可以关闭某些类型的通知）

## 团队协议

基于本次回顾的讨论，团队达成以下协议：

1. **状态同步协议**
   - 每次完成 Story 开发后，立即更新 sprint-status.yaml
   - 运行 `/bmad-sprint-status` 验证状态一致性
   - 将状态更新纳入 Definition of Done

2. **代码审查协议**
   - 每个 Story 完成后必须进行代码审查
   - 优先使用 `/bmad-code-review` 工作流
   - 审查发现的问题必须在 24 小时内修复

3. **数据库迁移协议**
   - 创建迁移文件前先检查现有版本号
   - 采用最大版本号 +1 的策略
   - 迁移文件命名遵循语义化原则

4. **技术债务管理协议**
   - 每个 Epic 预留 20% 时间用于偿还技术债务
   - 高优先级技术债务纳入下一个 Epic 的 Story
   - 每月进行一次技术债务审计

5. **API 契约协议**
   - 前后端开发前先评审 API 设计
   - 使用 Swagger/OpenAPI 定义接口契约
   - 建立 Mock Server 支持并行开发
